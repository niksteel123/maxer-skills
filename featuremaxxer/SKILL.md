# Featuremaxxer — Feature-to-Implementation Engine

> Transform a feature research/spec document into executed, tested code through a structured prompt pipeline — fast-tracked with no separate plan or TDD stage.

---

## The Pipeline

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   /featuremaxxer feature-doc.md                                         │
│       │                                                                 │
│       ├── Phase 0-1: Deep understand feature + codebase + tests         │
│       │                                                                 │
│       ├── Generates: FEATURE OPTIMIZATION    → featuremaxxer/active/    │
│       ├── Generates: BEADS CREATION          → featuremaxxer/active/    │
│       ├── Generates: BEADS OPTIMIZATION      → featuremaxxer/active/    │
│       ├── Generates: EXECUTION KICKOFF       → featuremaxxer/active/    │
│       └── Generates: EXECUTION PUSH          → featuremaxxer/active/    │
│                                                                         │
│   USER RUNS feature optimization → improves feature doc + produces      │
│       │                            implementation & testing strategy     │
│       │                                                                 │
│   USER RUNS beads creation → produces executable bead graph             │
│       │                                                                 │
│   USER RUNS beads optimization → audits and fixes bead graph            │
│       │                                                                 │
│   USER RUNS execution kickoff/push → agents implement the feature       │
│                                                                         │
│   (optional: run /harmonymaxer retroactively to sync layer docs)        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Why Fast-Tracked?

Featuremaxxer deliberately skips separate plan and TDD stages. The feature doc IS the plan — it defines what to build. The optimization handoff front-loads implementation strategy and testing strategy as deliverables, eliminating the need for intermediate planning documents. The beads encode both the implementation and its tests.

This works because:
- The feature doc already contains the architectural direction
- The optimization agent studies the codebase deeply and produces the strategy
- The strategy is a temporary artifact consumed during beads creation, then discarded
- Each bead embeds its own testing requirements (TDD-first, E2E-focused)

### File Structure

```
featuremaxxer/                           (lives at project root or docs/featuremaxxer/)
├── active/                              (current run — generated prompts)
│   ├── feature-optimization.md          (prompt to improve feature doc + produce strategy)
│   ├── implementation-strategy.md       (PRODUCED by optimization agent — temporary)
│   ├── beads-creation.md                (prompt to create beads from optimized doc + strategy)
│   ├── beads-optimization.md            (prompt to audit and fix bead graph)
│   ├── execution-kickoff.md             (prompt to onboard fresh execution agents)
│   └── execution-push.md               (prompt to push continuing execution agents)
├── archived/                            (previous runs — kept for reference)
│   ├── run-001/
│   └── run-002/
└── .run                                 (counter file: current run number)
```

Note: `implementation-strategy.md` is NOT generated by the skill — it is PRODUCED by the optimization agent when they run `feature-optimization.md`. It becomes an input to beads creation and is explicitly temporary.

### Pipeline Sequencing — Sequential Invocation

The 5 prompts have a **strict dependency chain**. Downstream prompts become fully project-specific only after upstream prompts produce their outputs:

```
/featuremaxxer (first run)
    │
    ├── feature-optimization.md    ← FULLY FILLED (needs only feature doc + codebase)
    │
    ├── beads-creation.md          ← SCAFFOLD (implementation-strategy.md doesn't
    ├── beads-optimization.md      ←   exist yet until optimization agent produces it.
    ├── execution-kickoff.md       ←   Marked with {AFTER_OPTIMIZATION} sections.
    └── execution-push.md          ←   Still 90% filled from Phase 1 analysis.)

User runs feature-optimization.md → produces improved doc + implementation-strategy.md

/featuremaxxer --beads-only (second run — reads strategy + optimized doc)
    │
    ├── beads-creation.md          ← NOW FULLY FILLED (knows testing strategy,
    └── beads-optimization.md      ←   sequencing, file surfaces)

User runs beads-creation → produces bead graph
User runs beads-optimization → audits beads

/featuremaxxer --execution-only (third run — reads bead graph)
    │
    ├── execution-kickoff.md       ← NOW FULLY FILLED (real bead IDs,
    └── execution-push.md          ←   file-surface maps, dependency spine)
```

## Phase 0: ORIENTATION

**Duration:** 5-10 minutes
**Goal:** Find the feature doc, the target codebase, and understand project structure
**Gate:** Know the feature doc, codebase layout, tech stack, and output location

### Protocol

#### Step 0.1: Read Foundation Documents

Read these if they exist (in order):
1. `AGENTS.md` — agent guidelines, toolchain, rules
2. `README.md` — project overview
3. `CLAUDE.md` — project-specific instructions

#### Step 0.2: Locate the Feature Document

The feature document is the research doc, feature spec, or design document that defines what to build. If the user specified a path, use that. Otherwise, scan:

```
SEARCH PATTERNS (priority order):
  docs/research/*research*.md, docs/research/*feature*.md
  docs/spec*.md, docs/feature*.md
  thoughts/research/*.md
  *-research.md, *-feature.md, *-spec.md
  docs/**/*.md (any large structured design doc)
```

The feature document is typically:
- A structured design/research document defining new capabilities
- Contains architectural decisions, data structures, technical analysis
- May include implementation strategy, phasing, or integration notes
- May range from ~200 lines (brief spec) to ~6000+ lines (deep research)

#### Step 0.2b: Classify the Feature Document

Feature doc size and depth directly affect downstream prompt content:

```
FEATURE DOC CLASSIFICATION:

  BRIEF SPEC (<500 lines):
    Signals: concise requirements, minimal analysis, few/no ADRs,
    may be bullet-point style or informal notes
    → Optimization handoff must push harder on independent thinking
    → More reasoning guidance needed — reviewer fills analytical gaps
    → Strategy deliverable is MORE important (spec may lack sequencing)
    → Beads will need more embedded context (less to reference)

  STRUCTURED SPEC (500-2000 lines):
    Signals: clear sections, some analysis, data structures defined,
    some decisions documented, moderate detail
    → Standard optimization handoff depth
    → Balanced reviewer independence + guided focus areas
    → Strategy can build on existing analysis
    → Beads can reference specific spec sections for context

  DEEP RESEARCH (2000+ lines):
    Signals: comprehensive analysis, many ADRs, state machines,
    detailed data schemas, multiple alternatives evaluated,
    formal invariants, extensive trade-off documentation
    → Optimization handoff focuses on cross-section consistency
    → Reviewer should look for tensions BETWEEN sections
    → Strategy extracts from existing analysis rather than creating
    → Beads benefit from rich source material but risk information overload
    → Risk: reviewer does surface-level pass on long doc — handoff must
      demand deep reading with specific section focus areas

  Classify and record. This affects Phase 2 prompt generation.
```

#### Step 0.3: Locate the Target Codebase and Existing Docs

Identify:
- **Codebase root**: Where is the source code?
- **Architecture docs**: Layer docs, plan docs, spec docs that describe the EXISTING system
- **Test suite**: Where do tests live? What frameworks? What patterns?
- **Build/test commands**: How to build, lint, test?

```
CODEBASE PATTERNS:
  src/, lib/, crates/, packages/        → Source code
  tests/, __tests__/, spec/, e2e/       → Test suites
  docs/strategy/, docs/layers/, thoughts/ → Architecture docs
  Cargo.toml, package.json, pyproject.toml → Build config
```

#### Step 0.4: Detect Project Domain

```
Same domain detection as other *maxer skills:
  FINANCIAL/DEFI, DISTRIBUTED SYSTEMS, WEB APPLICATION,
  PROTOCOL DESIGN, INFRASTRUCTURE

Additionally detect:
  TECH STACK: Rust/Anchor, TypeScript/React, Python, Go, etc.
  BUILD TOOL: cargo, bun/npm, pip, go build
  TEST FRAMEWORK: cargo test, jest/vitest, pytest, playwright
  LINT TOOL: clippy, eslint, ruff
```

#### Step 0.5: Detect Implementation State

```
IMPLEMENTATION STATE:
  Which layers/modules are already implemented?
  Which are partially implemented?
  Which are planned but not built?
  What's the testing infrastructure status?
  Are there E2E test harnesses or only unit tests?

  This directly affects:
  - What the feature can build on top of
  - What integration surfaces exist
  - What test infrastructure is available vs needs creation
```

#### Step 0.6: Detect Existing Pipeline State

Check if earlier pipeline outputs exist (for `--beads-only` and `--execution-only` re-runs):

```
EXISTING STRATEGY:
  featuremaxxer/active/implementation-strategy.md
  If found: read it. Downstream prompts can be FULLY FILLED.

EXISTING BEAD GRAPH:
  .beads/ directory or bead tool present
  If found: execution prompts reference real bead IDs and file surfaces.
```

#### Step 0.7: Identify Output Location

```
Location priority for featuremaxxer/ root:
  1. featuremaxxer/ (if exists)
  2. docs/featuremaxxer/ (if docs/ exists)
  3. Create featuremaxxer/ at project root
```

#### Step 0.8: Apply Re-Run Skip Logic

If invoked with `--beads-only` or `--execution-only`, most work is already done.
Skip accordingly:

```
RE-RUN ROUTING:

  --beads-only:
    REQUIRES: featuremaxxer/active/implementation-strategy.md exists
    If missing: ERROR — "Run feature optimization first to produce the strategy."

    SKIP: Phases 0-2 (orientation, deep understanding, optimization handoff)
    READ: implementation-strategy.md + optimized feature doc
    RESUME AT: Phase 3 (Beads Creation Handoff Generation)
    ALSO GENERATE: Phase 4 (Beads Optimization)

    The strategy doc contains the integration surface map, sequencing,
    testing landscape — everything Phase 1 would have produced. Don't redo it.

  --execution-only:
    REQUIRES: bead graph exists (.beads/ or equivalent)
    If missing: ERROR — "Run beads creation first to produce the bead graph."

    SKIP: Phases 0-4 (everything before execution prompts)
    READ: bead graph + feature doc + implementation-strategy.md (if still exists)
    RESUME AT: Phase 5 (Execution Prompt Generation)

    Extract real bead IDs, file-surface maps, and dependency spine from
    the actual bead graph. These replace the scaffolded placeholders from
    the first run.

  --optimization-only:
    SKIP: Phases 3-5 (everything after optimization handoff)
    GENERATE: Only Phase 2 (Feature Optimization Handoff)
    Useful when you only need the feature reviewed, not full pipeline.

  (no flag — full run):
    Execute all phases. Downstream prompts use scaffolds with
    {AFTER_OPTIMIZATION} / {AFTER_BEADS} placeholders where upstream
    outputs don't yet exist.
```

## Phase 1: DEEP UNDERSTANDING (ULTRATHINK REQUIRED)

**Duration:** 45-120 minutes depending on feature doc and codebase size
**Goal:** Deeply understand the feature, the existing codebase, the test landscape, and all integration points
**Gate:** Can articulate exactly how this feature integrates into the existing system, what code surfaces it touches, what tests exist, and what the implementation strategy should be

### This Phase Is Why Featuremaxxer Exists

Other *maxer skills work primarily with documentation. Featuremaxxer works at the intersection of documentation AND code. The optimization handoff must be informed by deep codebase understanding — a reviewer who doesn't know the actual code will produce generic feedback that misses integration-critical details. The beads must reference real files, real symbols, real tests.

### Protocol

#### Step 1.1: Read the Feature Document Fully

Read the ENTIRE feature doc. For large documents (>3000 lines), read in chunks but do NOT skip sections.

Build a mental model of:
- The core vision and value proposition
- New data structures, APIs, and state machines
- Architectural decisions (locked vs open)
- Implementation phases or strategies (if described)
- Dependencies on existing infrastructure
- Testing requirements (if described)

#### Step 1.2: Read the Existing Architecture Docs

Read ALL layer docs, plan docs, or architecture docs that describe the EXISTING system. The feature will integrate INTO this system — you must understand it deeply.

```
ARCHITECTURE CONTEXT:
  What is the existing system's overall architecture?
  What components/programs/services exist?
  What data structures and APIs are defined?
  What invariants must the feature respect?
  What decisions constrain the feature's design space?
  How does the feature relate to existing capabilities?
```

Note: Layer docs provide EXISTING SYSTEM context, NOT the feature itself (unless already harmonized). The feature doc is the authority for the new capability.

#### Step 1.3: Study the Codebase (CRITICAL — THIS IS UNIQUE TO FEATUREMAXXER)

Study the actual implementation to understand what exists today:

```
CODEBASE STUDY:

  INTEGRATION SURFACES:
    Which modules/files will the feature touch?
    What are the current types, functions, and APIs at those surfaces?
    What are the module boundaries and dependency directions?
    Which interfaces are stable vs actively changing?

  IMPLEMENTATION PATTERNS:
    What patterns does the codebase use? (error handling, async, state management)
    What shared utilities, traits, or abstractions exist?
    What naming conventions are used?
    What module structure conventions are used?

  CURRENT STATE vs FEATURE ASSUMPTIONS:
    Does the feature doc assume infrastructure that doesn't exist yet?
    Does the feature doc describe types that differ from what's implemented?
    Are there implementation details the feature doc doesn't account for?

  RELEVANT CODE SURFACES:
    List every file/module the feature will likely touch.
    For each: current purpose, key types/functions, likely changes.
    Group by: core changes vs supporting changes vs test changes.
```

#### Step 1.4: Study the Existing Test Suite (CRITICAL FOR FAST-TRACKING)

Since we skip separate TDD, the testing strategy must be built here:

```
TEST SUITE STUDY:

  TEST INFRASTRUCTURE:
    What test frameworks are used? (cargo test, jest, playwright, etc.)
    Are there E2E test harnesses? What do they test?
    Are there real-runtime test harnesses? (live server, live daemon, etc.)
    Are there integration test patterns? What do they look like?

  EXISTING TESTS AFFECTED BY FEATURE:
    Which existing tests touch integration surfaces?
    Which tests will break when the feature is implemented?
    Which tests need rewriting vs extending vs replacing?

  TEST GAPS:
    What test coverage exists at integration surfaces?
    Where are tests mock-heavy vs real-runtime?
    What E2E flows don't have test coverage?
    What would a comprehensive testing strategy require?

  TEST PATTERNS TO FOLLOW:
    Which existing test files are good quality exemplars?
    What helper/harness infrastructure exists?
    What assertion patterns are used?
```

#### Step 1.5: Map Integration Surfaces

Build the complete map of where the feature touches existing code:

```
INTEGRATION SURFACE MAP:

For each integration point:
  1. CODE LOCATION: File path, module, function/type
  2. CURRENT BEHAVIOR: What this code does now
  3. REQUIRED CHANGE: What the feature needs to change/add
  4. CHANGE COMPLEXITY: Trivial / Moderate / Complex / Risky
  5. DEPENDENCIES: What other integration points depend on this one
  6. TEST IMPACT: Which tests are affected
  7. RISK: What could go wrong

Group surfaces by:
  - Core feature implementation (the main new capability)
  - Supporting changes (existing code that must adapt)
  - Test changes (test files that must be updated/created)
  - Build/config changes (if any)
```

#### Step 1.6: Identify Non-Negotiables

Extract locked architectural decisions that the feature must respect:

```
NON-NEGOTIABLES:

  FROM EXISTING ARCHITECTURE:
    Which design decisions are locked? (consensus, data model, security model, etc.)
    Which invariants must be preserved?
    Which interfaces are frozen?

  FROM FEATURE DOC:
    Which feature decisions are locked by the author?
    Which are explicitly marked as non-negotiable?
    Which tradeoffs has the author already resolved?

  COMBINED:
    List all non-negotiables with source and rationale.
    These go directly into the optimization handoff's "Don't Question These" section.
```

#### Step 1.7: Build Testing Landscape

Synthesize Steps 1.4 and 1.5 into a testing strategy outline:

```
TESTING LANDSCAPE:

  EXISTING TESTS TO UPDATE:
    - {test_file}: {what needs changing} ({reason})
    - ...

  NEW TESTS TO CREATE:
    - {test_description}: {what it validates} ({type: unit/integration/E2E})
    - ...

  E2E / REAL-RUNTIME TESTS NEEDED:
    - {flow_description}: {what runtime infrastructure is required}
    - ...

  TEST INFRASTRUCTURE GAPS:
    - {gap}: {what needs to be created}
    - ...

  TESTING GATES (for beads):
    - After {milestone}: {which tests must pass}
    - ...
```

#### Step 1.8: Identify Implementation Strategy Candidates

Based on all analysis, determine the optimal implementation sequencing:

```
IMPLEMENTATION STRATEGY:

  DEPENDENCY SPINE:
    What must be built first? (foundational types, core logic)
    What can be built in parallel? (independent surfaces)
    What must come last? (integration tests, E2E gates)

  PARALLEL TRACKS:
    Which implementation surfaces can be worked on simultaneously?
    What are the file-ownership boundaries between parallel tracks?
    What coordination points exist between tracks?

  SEQUENTIAL CONSTRAINTS:
    Which changes must land before others?
    What are the critical-path items?
    Where do test rewrites gate implementation?

  BEAD TOPOLOGY SKETCH:
    Root epic: {feature name}
    ├── Epic: {foundational work}
    ├── Epic: {core implementation track 1}
    ├── Epic: {core implementation track 2}
    ├── Epic: {test rewrite and expansion}
    ├── Epic: {E2E / real-runtime harnesses}
    └── Epic: {final QA gates}
```

#### Step 1.9: Map Architecture Tensions (CRITICAL — drives optimization depth)

Tensions are NOT risks. Risks are "what could go wrong." Tensions are conflicts between LEGITIMATE goals that the feature must resolve. These are the most valuable things to surface for the optimization agent because they drive the deepest insights.

```
TENSION MAPPING:

  FEATURE vs EXISTING ARCHITECTURE:
    Where does the feature's model conflict with patterns the existing
    system already uses? (e.g., feature wants event-driven but system
    is request-response; feature wants shared state but system is isolated)
    For each: what's the tension? How does the feature doc resolve it?
    Could the resolution be wrong?

  FEATURE INTERNAL TENSIONS:
    Where does the feature want two things that conflict?
    (e.g., simplicity vs completeness; performance vs safety;
    flexibility vs type safety; fast shipping vs clean architecture)
    For each: what tradeoff did the feature doc choose? Is it the right one?

  FEATURE vs DEPLOYMENT REALITY:
    Where does the feature assume conditions that may not hold?
    (e.g., assumes low latency but system is distributed; assumes
    atomic operations but infrastructure doesn't support them)

  ALTERNATIVE APPROACHES NOT CHOSEN:
    What other approaches could achieve the same outcome?
    Why did the feature doc choose this one?
    What would a skeptic say is wrong with this approach?
    What would a competitor's architect do differently?

  For each tension:
    - TENSION: {what conflicts with what}
    - CURRENT RESOLUTION: {how the feature doc handles it, or doesn't}
    - WHY IT MIGHT BE WRONG: {the uncomfortable question}
    - IMPLICATION FOR BEADS: {how this affects implementation sequencing}
```

These tensions go directly into the optimization handoff as the seed material for the "Think Independently" section. They're what distinguish deep architectural review from surface-level checklist verification.

#### Step 1.10: Identify Risks

```
RISK IDENTIFICATION:

  TECHNICAL RISKS:
    Where might the feature doc's assumptions not match reality?
    Where is the implementation complexity likely underestimated?
    What cross-cutting concerns might the feature create?

  INTEGRATION RISKS:
    Which integration surfaces are highest-risk?
    Where might breaking changes cascade?
    What backwards-compatibility concerns exist?

  TESTING RISKS:
    Where is test infrastructure insufficient?
    What flows are hard to test?
    Where might mock-only testing miss real bugs?
```

## Phase 2: FEATURE OPTIMIZATION HANDOFF GENERATION

**Duration:** 30-60 minutes
**Goal:** Generate a comprehensive prompt that deeply reviews and improves the feature doc, and produces an implementation + testing strategy
**Gate:** Prompt produces both an improved feature doc AND a temporary strategy doc

This is the MOST IMPORTANT generated artifact. The optimization handoff must produce two outputs:
1. **Improved feature doc** — edited in place with targeted improvements
2. **Implementation & testing strategy** — saved to `featuremaxxer/active/implementation-strategy.md` (temporary, consumed by beads creation)

### What the Feature Optimization Handoff Must Contain

See **references/TEMPLATES.md -> Feature Optimization Template** for the full structure.

```
1. MISSION STATEMENT
   Frame the agent as a deep architecture reviewer AND implementation strategist.
   Two deliverables: improved feature doc + implementation strategy.
   "You are NOT implementing anything. You are reading, analyzing, and improving."

2. GOVERNING PRINCIPLE
   Clean architecture from day zero (or appropriate project context).
   What refactoring constraints apply? (zero users = refactor freely,
   production system = careful migration)
   "If you find a simpler, more correct way — propose it."

3. SOURCE-OF-TRUTH HIERARCHY
   Feature doc > layer docs for the feature domain.
   Layer docs = existing system context.
   Codebase = ground truth for what exists today.
   "If feature doc and layer docs contradict on the feature, feature doc wins."

4. WHAT TO READ (comprehensive, with reading strategy)
   Feature doc: path, line count, key sections to focus on.
   Architecture docs: paths, what to focus on (integration surfaces).
   Codebase: specific files/modules with reading strategy (don't read everything).
   Test suite: specific test files and harnesses to study.

5. THINK INDEPENDENTLY DIRECTIVE (CRITICAL — NOT A PRESCRIBED CHECKLIST)
   "After reading everything, your process should be:
   1. Absorb the architecture. Let it marinate.
   2. Generate YOUR OWN questions — the uncomfortable ones.
   3. Research externally where YOUR analysis identifies gaps.
   4. Think about what's missing entirely.
   5. Challenge the approach itself — what would an adversary architect do differently?"
   These docs have been through reviews. Surface-level issues are fixed.
   What remains are deep, non-obvious insights.
   "Trust your instincts — if something feels off, chase that feeling."

   SEED WITH TENSIONS FROM PHASE 1 (Step 1.9):
   Inject the mapped tensions directly into this section. Format each as:
   "The feature doc resolves {TENSION} by choosing {RESOLUTION}.
    Consider whether this is actually correct. What if {WHY_IT_MIGHT_BE_WRONG}?"
   These are NOT questions for the reviewer to answer verbatim — they're
   thinking catalysts that point at the deepest architectural fault lines.
   The reviewer may discover the tensions are correctly resolved (good) or
   may find that the resolution creates downstream problems (better — that's
   the insight we need).

6. NON-NEGOTIABLES (from Phase 1 Step 1.6)
   Locked decisions the reviewer must NOT question.
   Organized by category (core paradigm, infrastructure, constraints).
   "Everything else is fair game for questioning."

7. IMPROVEMENT DIRECTIVES
   What to do with the feature doc:
   - Strengthen weak technical sections
   - Add missing risk/trade-off analysis
   - Correct technical errors
   - Tighten verbose prose
   - Ensure self-consistency
   What NOT to do:
   - Don't rewrite strong sections for the sake of it
   - Don't add fluff
   - Don't make code changes
   - Don't change non-negotiables

8. IMPLEMENTATION STRATEGY DELIVERABLE (CRITICAL — fast-tracking requirement)
   The optimization agent must ALSO produce:
   `featuremaxxer/active/implementation-strategy.md` containing:
   - Implementation surface map (which files/modules the feature touches)
   - Sequencing analysis (what must come first, what can parallelize)
   - Dependency spine
   - Testing strategy:
     * Existing tests to update (specific file paths and what changes)
     * New tests to create (descriptions and types)
     * E2E / real-runtime gates
     * Test infrastructure gaps to fill
   - Risk assessment
   This is explicitly temporary — consumed during beads creation, then discarded.

9. WHAT A GREAT OUTCOME LOOKS LIKE
   Not generic praise. Not restating the docs in different words.
   "Questions nobody has asked yet."
   "Connections nobody has made yet."
   "Risks nobody has identified yet."
   "Simplifications nobody has considered yet."
   "Cross-document inconsistencies nobody has noticed yet."

10. FOLLOW-UP FRESH-EYES PASS
    Re-read feature doc + implementation strategy with fresh eyes.
    Verify strategy is implementable against actual codebase.
    Verify testing strategy covers all integration surfaces.
    Verify no non-negotiable was accidentally violated.
```

### What Makes a Great Feature Optimization Handoff

```
GREAT feature optimization handoffs:
  ✓ Include specific codebase files/modules for the reviewer to study
  ✓ Non-negotiables are explicit and comprehensive
  ✓ "Think independently" directive (not prescribed checklist)
  ✓ Implementation strategy is a REQUIRED deliverable (not optional)
  ✓ Testing strategy front-loads the heavy analytical work
  ✓ Success criteria expect original insights, not surface tweaks
  ✓ Source-of-truth hierarchy is clear

BAD feature optimization handoffs:
  ✗ "Review this feature doc" with no codebase context
  ✗ Prescribed checklist instead of independent thinking
  ✗ No implementation strategy requirement (leaves it to beads creation)
  ✗ No testing strategy (beads agent has to redo test analysis)
  ✗ Generic success criteria ("improve the doc")
  ✗ No non-negotiables (reviewer wastes time questioning locked decisions)
```

## Phase 3: BEADS CREATION HANDOFF GENERATION

**Duration:** 20-30 minutes
**Goal:** Generate a prompt that converts the optimized feature doc + implementation strategy into executable beads
**Gate:** Prompt produces self-contained, TDD-first beads with embedded testing requirements

### What the Beads Creation Handoff Must Contain

See **references/TEMPLATES.md -> Beads Creation Template** for the full structure.

```
1. MISSION
   Frame: "converting architecture into executable work orders"
   Self-contained beads — agent implements from bead alone.
   "The agent implementing a bead should NOT need to read the feature doc."
   Cross-cutting: the feature likely spans multiple modules/layers.

2. IMPLEMENTATION CONTEXT
   What's implemented vs planned.
   Practical consequence: don't build against imaginary infrastructure.

3. MANDATORY PRE-BEAD RESEARCH
   Read the optimized feature doc fully.
   Read the implementation strategy fully.
   Study the codebase integration surfaces.
   Study the existing test suite.
   Produce a private implementation map BEFORE creating beads:
   - what can be parallelized
   - what must be sequential
   - which tests must be rewritten
   - which new tests must be added
   - which real-runtime harnesses exist vs must be created

4. BEAD CREATION GOALS
   Self-contained. Implementation-ready. Dependency-correct.
   Parallelized where safe. TDD-first. Architecture-preserving.
   Test-harness-aware (real-runtime, not just mocks).

5. SUGGESTED BEAD HIERARCHY (from Phase 1 Step 1.8)
   Root epic with sub-epics based on implementation tracks.
   Sequential vs parallel tracks identified.
   "Explicitly encode parallel/sequential decisions into dependencies."

6. BEAD QUALITY STANDARD
   Minimum contents: background, current-state context, target behavior,
   implementation detail (exact files/types/functions), testing detail
   (tests to rewrite, tests to add, real-runtime validation), acceptance
   criteria, dependencies.
   "Never say 'see the feature doc' — embed the context directly."

7. TESTING STRATEGY MUST BE EMBEDDED IN BEADS
   Every bead touching the feature must carry testing requirements.
   Specify: which tests to rewrite, new tests to add, regression gates,
   real-runtime validation expectations.
   "Mock/unit tests are useful but not authoritative for runtime-facing work."

8. TOOLING (if bead tool is available)
   Commands for creating, updating, showing beads.

9. WHAT A GREAT OUTCOME LOOKS LIKE
   Bead tree mirrors the feature architecture cleanly.
   Foundational work sequenced before dependent work.
   Every bead is self-contained with embedded testing requirements.
   "The implementation session can execute without reopening the feature doc."

10. FOLLOW-UP FRESH-EYES PASS
    Re-read feature doc + every bead with fresh eyes.
    Self-containment check. Dependency correctness check.
    Testing completeness check. File/symbol accuracy check.
```

## Phase 4: BEADS OPTIMIZATION GENERATION

**Duration:** 15-20 minutes
**Goal:** Generate a meticulous bead-by-bead QA audit prompt
**Gate:** Prompt covers all audit dimensions with project-specific checks

### What the Beads Optimization Must Check

See **references/TEMPLATES.md -> Beads Optimization Template** for the full structure.

```
1. SELF-CONTAINMENT AUDIT
   Could an agent implement without reading the feature doc?
   Exact files/symbols? Exact tests? Real-runtime signoff expectations?

2. DEPENDENCY AUDIT
   Correct sequencing? Safe parallelism? Missing/fake dependencies?

3. TECHNICAL ACCURACY AUDIT
   File paths real? Symbol names real? Current code described accurately?
   Feature doc's architecture reflected accurately?

4. TESTING AUDIT
   Existing tests to rewrite named explicitly?
   New tests named explicitly enough?
   Real-runtime acceptance required where appropriate?
   Not accidentally relying on mock-only coverage?

5. PLATFORM / SURFACE COVERAGE AUDIT
   All integration surfaces covered?
   All platforms/modules represented?
   (Project-specific coverage items from Phase 1)

6. COVERAGE MATRIX AUDIT
   Feature subsystem → bead mapping
   Integration surface → bead mapping
   Test requirement → bead mapping
   Architecture decision → bead(s) that implement it

7. FIX AND FOLLOW-UP FRESH-EYES PASS
   Fix thin, vague, or dependency-confused beads.
   "If two different agents could read the same bead and make materially
   different implementation decisions, that bead is not ready."
```

## Anti-Patterns

### The Generic Review
```
BAD: "Review this feature doc and improve it" with no codebase context.

WHY: Without knowing the actual code, the reviewer can't assess
implementation feasibility, identify integration risks, or produce
a useful testing strategy. The optimization handoff MUST include
specific codebase files to study.
```

### The Checklist Trap
```
BAD: Prescribed checklist of things to verify instead of independent thinking.

WHY: The feature doc has likely been through reviews already. What it
needs is fresh-eyes INDEPENDENT thinking — questions nobody asked,
connections nobody made, risks nobody identified. A checklist produces
surface-level validation, not deep insight.
```

### The Mock-Only Mirage
```
BAD: Beads include testing requirements but only specify unit tests and mocks.

WHY: For runtime-facing work, mock-only tests are canaries, not proof.
Real acceptance requires E2E tests against actual running infrastructure.
Every bead touching integration surfaces must specify real-runtime
validation, not just mock coverage.
```

### The Strategy Vacuum
```
BAD: Beads creation without an implementation strategy — the beads agent
must figure out sequencing, parallelization, and testing from scratch.

WHY: This is why featuremaxxer front-loads the strategy into the
optimization handoff. The optimization agent has deep codebase context
and produces the strategy as a deliverable. The beads agent CONSUMES
the strategy, they don't create it.
```

### The Context Leak
```
BAD: Beads say "see the feature doc for details" instead of embedding context.

WHY: Execution agents implement from beads, not from feature docs. Every
bead must be self-contained — exact files, exact types, exact tests,
exact acceptance criteria. "See the doc" is a planning failure.
```

### The Silo Split
```
BAD: Implementation tracks treated as independent when they share integration
surfaces, causing merge conflicts and inconsistent implementations.

WHY: Parallel tracks must have clear file-ownership boundaries. Shared
surfaces need explicit coordination points. The bead dependency graph
must encode these constraints, not leave them implicit.
```

### The Deferred Testing Trap
```
BAD: "We'll write tests after the implementation is done."

WHY: TDD-first means the tests are part of the bead, not a follow-up.
Every implementation bead must include testing as acceptance criteria.
Deferring tests to a separate "testing phase" creates a gap where
broken behavior ships without being caught.
```

### The Harmonization Prerequisite
```
BAD: Refusing to create beads because layer docs haven't been harmonized
with the feature yet.

WHY: Featuremaxxer is designed to run PARALLEL to or BEFORE harmonymaxer.
The feature doc is the source of truth for the feature. Layer docs
provide existing system context. The beads reference the feature doc
directly — they don't require pre-harmonized layer docs.
```
