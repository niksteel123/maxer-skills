# Codebase-to-Spec Output Template

> This is the unified output template for the `codebase-to-spec` skill.
> The skill reverse-engineers an existing codebase into a comprehensive specification document
> covering system architecture, functional requirements, and design elevation opportunities.
>
> The output is a single document with 19 sections organized in three parts:
> - **Part I** (sections 1-11): System Architecture — what the system IS
> - **Part II** (sections 12-15): Functional Specification — what the system DOES
> - **Part III** (sections 16-19): Design Elevation — what the system SHOULD BECOME
>
> Every claim in the spec must trace back to source code with file paths and line ranges.
> The document serves as the canonical input to the downstream pipeline:
> `codebase-to-spec` --> `spec-to-tdd` --> `spec-to-strategy` --> `beads-from-sources` --> `beads-to-execution`

---

## Output Filename Pattern

```
{project_root}/docs/specs/{YYYY-MM-DD}-{slug}-extracted-spec.md
```

If the directory `{project_root}/thoughts/shared/specs/` exists, use that instead:

```
{project_root}/thoughts/shared/specs/{YYYY-MM-DD}-{slug}-extracted-spec.md
```

Examples:
- `docs/specs/2026-02-24-auth-service-extracted-spec.md`
- `thoughts/shared/specs/2026-02-24-mesh-transport-extracted-spec.md`

The `{slug}` should be a kebab-case identifier for the codebase or subsystem being specified
(e.g., `payment-gateway`, `event-sourced-kernel`, `agent-mesh-transport`).

## Section Guidelines

### Part I - System Architecture

#### Section 1: System Overview
- Lead with the ASCII architecture diagram — this orients all readers
- Every component in the diagram MUST appear in the component descriptions table
- Layer boundaries should describe what crosses them and what does not
- Technology stack must include versions observed in lockfiles/configs

#### Section 2: Data Flow Diagrams
- One diagram per major user-facing flow (auth, CRUD, real-time sync, etc.)
- Use ASCII sequence diagrams with clear actor labels
- Every flow must cite Source Evidence with file:line references
- Include a summary table at the end for quick scanning

#### Section 3: Data Models
- Extract every domain model with field-level detail (type, constraints, defaults)
- Include relationship diagrams showing cardinality
- Document invariants (business rules) observed in validation code
- Event/message types must include producer and consumer mappings

#### Section 4: API Surface
- Every endpoint needs method, path, auth requirement, request/response examples, and error codes
- WebSocket protocols need message type tables with direction (C->S, S->C)
- CLI interfaces need argument and flag documentation
- Internal module APIs need exported function signatures and usage examples

#### Section 5: Component Map
- One entry per logical component (not per file)
- Dependencies and dependents create a bidirectional graph — no orphans
- Include the ASCII dependency graph for visual orientation
- Key files table helps implementers know where to look first

#### Section 6: State Management
- Identify the state management approach (centralized store, event-sourced, distributed)
- Extract the full state shape from type definitions
- State machines MUST have transition tables AND ASCII diagrams
- Guards on transitions are critical — do not omit them

#### Section 7: Error Architecture
- Extract the error type hierarchy from the codebase
- Document propagation patterns (how errors flow between layers)
- Recovery strategies must be specific (retry count, backoff, fallback behavior)

#### Section 8: Security Architecture
- Document authn mechanism with flow diagram
- Document authz model with role/permission matrix
- Data protection must cover at-rest, in-transit, and input sanitization
- Secret management approach must be documented

#### Section 9: External Dependencies & Services
- Runtime deps from lockfiles with version, purpose, license, and criticality
- External services need connection details, auth, failure handling
- Risk assessment for stale or deprecated dependencies

#### Section 10: Operational Architecture
- Logging framework and structured field conventions
- Metrics with types (counter/gauge/histogram) and label dimensions
- Health check endpoints and what they validate
- Build targets and deployment configuration

#### Section 11: IADRs
- Minimum 3 IADRs for standard mode, 5+ for deep mode
- Each IADR must have Evidence with file:line references and usage counts
- Confidence must be honestly assessed — LOW is acceptable when evidence is thin
- Analysis section should explain the reasoning, not just state the decision

### Part II - Functional Specification

#### Section 12: Goals & Non-Goals
- Goals are inferred from what the system DOES (with evidence)
- Non-goals are inferred from what the system explicitly DOES NOT DO
- Every goal needs cross-referenced evidence (README, tests, implementation)
- Confidence levels distinguish "definitely a goal" from "probably a goal"

#### Section 13: Functional Requirements
- THE most important section — downstream pipeline depends entirely on FR quality
- Every FR MUST have: title, requirement statement, source evidence, tag, acceptance criteria, core test
- [ELEVATED] FRs MUST have elevation justification explaining why spec improves on code
- Source Evidence must use `file_path:line_range` format — no vague references
- Acceptance criteria use GIVEN/WHEN/THEN format
- Core tests must be executable pseudo-code (not just descriptions)
- Summary table at the end with tag distribution statistics

#### Section 14: User Stories
- Reconstruct user stories from handler code, UI components, and CLI commands
- Each story needs AS/I WANT/SO THAT format
- Acceptance criteria must be user-visible outcomes (not implementation details)
- Cross-reference to related FRs

#### Section 15: E2E Test Scenarios
- Clean slate protocol defines the base state for all scenarios
- Each scenario needs setup, execution steps, and verification assertions
- Evidence must trace back to the code paths being tested
- Coverage matrix maps scenarios to FRs and user stories

### Part III - Design Elevation

#### Section 16: Design Improvements Register
- Improvements are proposals for how the SPEC can be better than the CODE
- Each improvement needs current behavior (with source) and proposed behavior
- Migration path is critical — how to get from current to proposed
- Priority and effort estimates help downstream planning

#### Section 17: Technical Debt Register
- Concrete, actionable debt items — not vague complaints
- Each item needs type classification, severity, location, and remediation steps
- Evidence must show the specific code exhibiting the debt

#### Section 18: Reconciliation Notes
- Honest assessment of extraction quality per section
- Ambiguities MUST be surfaced, not hidden — downstream consumers need to know
- Human review items identify where AI inference is insufficient
- Extraction limitations describe what could NOT be analyzed and why

#### Section 19: Risks, Dependencies & Future Considerations
- Risks with likelihood/impact/mitigation
- Blocking dependencies that affect implementation order
- Future considerations for scaling, migration, or deprecation

### Mode Definitions

- **Surface Mode**: Quick overview extraction. All 19 sections present but at minimum depth.
  High-level structure, key APIs, and critical data models only. No deep flow tracing.
  Suitable for very large codebases (>500 files), initial orientation, or time-constrained analysis.
  Aims for 500+ line output.

- **Standard Mode**: Default extraction. Covers all 19 sections at the minimum quality bar.
  Suitable for codebases up to ~50K LOC or when time-constrained. Aims for 3,000+ line output.

- **Deep Mode**: Exhaustive extraction. Every section is populated to maximum depth. All flows
  diagrammed, all models documented field-by-field, all endpoints with full request/response
  examples. Suitable for critical systems or pre-rewrite analysis. Aims for 5,000+ line output.
